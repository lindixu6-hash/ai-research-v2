# AI搜索产品竞品分析

## 一、前置理论

### 1.1 AI 搜索的本质

AI 搜索本质上就是 **RAG（检索增强生成）**，只不过检索不在内部知识库，而是在外部搜索引擎：

```
用户提问 → AI 拆解关键词 → 调用搜索引擎 → 获取结果 → 拼接上下文 → AI 生成答案
```

所有 AI 搜索产品都是在这个基础流程上增加 **Agent 能力**：
- AI 根据输入生成多组检索词
- 拿到结果后审视质量
- 质量不行就再生成一组关键词，循环搜索
- 最后汇总所有结果生成分析报告

### 1.2 AI 搜索的两大核心价值

| 价值 | 目标用户 | 解决的问题 |
|------|----------|------------|
| **降低信息加工成本** | 咨询顾问、品牌从业者、内容创作者 | 节省人工阅读文章、汇总信息的时间 |
| **帮助不会搜索的人** | 普通用户（更大群体） | 不会抽象关键词、不懂搜索语法（AND/OR/-排除） |

**关键洞察**：AI 搜索本质上是「搜索最佳实践的产品化」——帮用户拆解关键词、组合搜索语句、多轮验证。

### 1.3 用户分层与需求差异

| 用户类型 | 代表产品 | 需求特点 |
|----------|----------|----------|
| **浅层用户** | 秘塔、Deepseek | 快速获取信息概览，了解知识面 |
| **深度用户** | Kimi、Wisper | 高质量结果，愿意花时间深入研究 |

**重要认知**：真正做研究时，用户会针对某个关键词花一上午深入研究——边搜边学边发现新关键词，这个过程太快了反而没用。

## 二、竞品分析

### 2.1 秘塔搜索

**定位**：国内较早的 AI 搜索产品，学术检索导向

**核心机制**：
```
拆关键词 → 搜索 → 生成报告
```

**特点**：

- 接入学术数据库，适合学术检索
- 主要用户：高校老师、学生
- 生成结构化研究报告

**优点**：
- 快速铺开知识面，适合综述阶段

**缺点**：

- 交互体验糟糕（复杂的前端展示）
- 对深度调研用户来说太粗糙
- 无法做到深入研究

**适合场景**：写论文时快速了解某个领域的研究现状

---

### 2.2 Kimi 研究模式

**定位**：多步推理型 AI 搜索

**核心机制**：
```
拆关键词 → 搜索 → 验证结果 → 不够？再搜 → 循环直到满足或到达上限
```

**特点**：

- 使用浏览器真正深入浏览网页（不只是看摘要）
- 模拟真实研究流程：搜索 → 发现新关键词 → 再检索
- 有向用户澄清问题的机制

**优点**：

- 流程最接近真实研究过程
- 会真正阅读网页内容，质量较高

**缺点**：

- 澄清问题有时候多余（明明已经很清楚了）
- 流程太快，无法深入研究
- 到达 action 上限（25/50步）就停止

**适合场景**：需要比普通搜索更深入，但不想花太多时间的场景

---

### 2.3 Deepseek 搜索

**定位**：极简 AI 搜索

**核心机制**：
```
拆关键词 → 搜索 → 直接生成答案
```

**特点**：

- 一步到位，快速响应
- 基于搜索摘要生成答案

**优点**：

- 速度最快
- 体验简洁

**缺点**：

- 只能通过摘要干活，无法深入阅读
- 结果质量依赖搜索 API 返回的摘要质量

**适合场景**：快速获取信息概览

---

### 2.4 Consensus

**定位**：学术论文多步推理搜索

**核心机制**：
```
拆关键词 → 检索论文 → 返回标题/作者/DOI → 多步循环
```

**特点**：

- 多步推理，检索步数和结果更多
- API 返回：标题、作者、DOI ID、期刊等

**优点**：
- 专注学术论文
- 检索结果较多

**缺点**：

- API 返回信息有限（无正文）
- 前端体验差，容易卡死

**适合场景**：学术论文检索

---

### 2.5 Aminer

**定位**：多 Agent 协作搜索

**核心机制**：
```
规划专家 → 拆分任务 → 多个调研员并行搜索 → 汇总生成报告
```

**特点**：

- 规划专家 + 多个调研员并行工作
- 类似 Subagent 模式

**优点**：
- 并行效率高
- 展示 Agent 协作过程

**缺点**：

- 不进行结果筛选，引入大量脏数据
- 前端必须在线，关闭页面任务中断
- API 返回信息有限，无法深入阅读论文

**适合场景**：演示 Agent 协作能力

---

### 2.6 Wisper Paper

**定位**：带验证机制的论文搜索

**核心机制**：
```
验证关键词 → 验证搜索结果 → 一篇篇检查 → 筛选 → 展示
```

**特点**：

- **不直接总结，而是做验证**
- 验证关键词 + 验证搜索结果
- 一篇篇进去看文章是否靠谱
- 有筛选机制（如搜索 167 篇 → 验证后筛出 93 篇）

**优点**：

- 结果质量最高
- 有明确的筛选机制和统计数据

**缺点**：

- 不帮助总结，需要用户自己阅读

**适合场景**：研三/博士生需要高质量结果时

## 四、行业共性问题

### 4.1 五大痛点

| 痛点 | 描述 | 影响 |
|------|------|------|
| **脏数据问题** | 大部分产品不筛选结果，把所有检索结果塞给模型 | 影响最终生成内容质量，可能产生幻觉 |
| **流程太快** | 无法像真实研究那样深入探索 | 只能做到综述，无法深度研究 |
| **前端体验差** | 复杂交互让用户头大，甚至容易卡死 | 用户体验糟糕 |
| **无法离线** | 部分产品前端必须在线，关闭页面任务中断 | 使用受限 |
| **结果质量不稳定** | 依赖搜索 API 返回的摘要质量 | 可能生成错误答案 |

### 4.2 根本原因分析

**大部分 AI 搜索产品实现逻辑非常简单**，就是：

1. 拆关键词
2. 调用搜索 API
3. 把结果塞给模型
4. 生成报告

真正复杂的地方在于：

- **结果筛选**：Wisper 是少数做筛选的产品
- **深入阅读**：Kimi 会用浏览器真正读网页
- **多轮验证**：避免脏数据进入上下文

## 五、产品机会

### 5.1 市场空白

基于竞品分析，存在以下机会：

1. **平衡型产品**：介于极简和深度之间，兼顾速度和质量
2. **兜底机制**：大部分产品没有搜索失败的兜底方案
3. **开源可部署**：大部分产品闭源，用户无法自部署
4. **学术 + 通用**：单一场景产品较多，全场景产品少

### 5.2 设计原则

基于竞品分析，一个好的 AI 搜索产品应该：

1. **有兜底机制**：搜索失败时仍能返回有用信息
2. **结果筛选**：验证搜索结果质量，过滤脏数据
3. **流式输出**：实时展示搜索进度，让用户看到思考过程
4. **适度深入**：不要太快，也不要太慢，平衡速度和质量
5. **简洁 UI**：避免复杂交互，降低使用门槛
6. **来源可追溯**：所有答案都标注来源

## 六、我的项目定位

### 6.1 项目目标

做一个**平衡型 AI 搜索产品**，介于极简和深度之间：

| 维度 | 定位 |
|------|------|
| 速度 | 适中，不过快也不过慢 |
| 质量 | 有结果筛选，过滤脏数据 |
| 体验 | 流式输出 + 简洁 UI |
| 部署 | 开源，可自部署 |

### 6.2 核心差异化

| 功能 | 竞品 | 我的方案 |
|------|------|----------|
| 兜底机制 | 大部分无 | 三层兜底：Tavily → DuckDuckGo → 兜底结果 |
| 结果筛选 | 少数有（Wisper） | 有 |
| 重试机制 | 少数有 | 指数退避重试 |
| 缓存 | 少数有 | 5 分钟内存缓存 |
| 开源 | 大部分闭源 | 开源，可自部署 |

### 6.3 技术选型

| 模块 | 技术方案 | 理由 |
|------|----------|------|
| 搜索 API | Tavily + DuckDuckGo | Tavily 质量高，DuckDuckGo 兜底 |
| AI 模型 | Moonshot/Kimi | 中文理解能力强，长上下文 |
| 前端 | React + Vite | 现代化、开发体验好 |
| 后端 | Node.js + Express | 轻量、易于部署 |
| 流式输出 | SSE | 实时性好，比 WebSocket 简单 |
