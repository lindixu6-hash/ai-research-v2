# AI 搜索项目竞品分析

## 一、竞品概览

### 1. 秘塔搜索 (Metaso)
**定位**：国内较早的 AI 搜索产品，学术检索导向

**特点**：
- 接入学术数据库，适合学术检索
- 主要用户：高校老师、学生
- 生成结构化研究报告

**痛点**：
- 交互体验糟糕（复杂的前端展示，让用户头大）
- 对于深度调研用户来说太粗糙，无法做到深入研究

---

### 2. Kimi 研究模式
**定位**：多步推理型 AI 搜索

**特点**：
- 一步输出 → 结果审视 → 不行再搜索（循环机制）
- 使用浏览器真正深入浏览网页（不只是看摘要）
- 有向用户澄清问题的机制
- 模拟真实研究流程：搜索 → 发现新关键词 → 再检索

**痛点**：
- 澄清问题有时候多余（明明已经很清楚了）
- 流程太快，无法深入研究
- 到达 action 上限就停止，无法持续深入

---

### 3. Deepseek 搜索
**定位**：极简 AI 搜索

**特点**：
- 直接输出结果，一步到位
- 基于搜索摘要生成答案

**痛点**：
- 只能通过摘要干活，无法深入阅读
- 结果质量依赖搜索摘要质量

---

### 4. Consensus (学术搜索)
**定位**：学术论文多步推理搜索

**特点**：
- 多步推理，检索步数和结果更多
- API 返回：标题、作者、DOI ID、期刊等

**痛点**：
- API 返回信息有限（无正文）
- 前端体验差，容易卡死

---

### 5. Aminer 多 Agent 协作
**定位**：多模型协作搜索

**特点**：
- 规划专家 + 多个调研员并行工作
- 最后汇总生成报告

**痛点**：
- **不进行结果筛选**，引入大量脏数据
- 前端必须在线，关闭页面任务中断
- API 返回信息有限，无法深入阅读论文

---

### 6. Wisper Paper
**定位**：带验证机制的论文搜索

**特点**：
- **不直接总结，而是做验证**
- 验证关键词 + 验证搜索结果
- 一篇篇进去看文章是否靠谱
- 有筛选机制（如搜索 167 篇 → 验证后筛出 93 篇）

**痛点**：
- 不帮助总结，需要用户自己阅读

---

## 二、AI 搜索产品的核心价值

### 价值一：降低信息加工成本
**目标用户**：咨询顾问、品牌从业者、内容创作者

**场景**：针对某个选题/关键词，需要大量搜集信息、汇总合并、输出完整报告

**价值**：用模型的文字理解能力，节省人工阅读文章的时间

---

### 价值二：帮助不会搜索的人
**目标用户**：大部分普通用户（更大群体）

**痛点**：
- 不会抽象关键词
- 不会组合关键词
- 不懂搜索语法（AND、OR、-排除等）

**价值**：AI 帮助拆解关键词、生成检索语句、多轮搜索

---

## 三、行业共性问题（痛点总结）

| 痛点 | 描述 | 影响 |
|------|------|------|
| **脏数据问题** | 大部分产品不进行结果筛选，把所有检索结果和摘要都塞给模型 | 影响最终生成内容质量 |
| **流程太快** | 无法像真实研究那样，针对某个关键词花一上午深入研究 | 只能做到综述，无法深入 |
| **前端体验差** | 复杂交互让用户头大，甚至容易卡死 | 用户体验糟糕 |
| **无法离线** | 部分产品前端必须在线，关闭页面任务中断 | 使用受限 |
| **结果质量不稳定** | 依赖搜索 API 返回的摘要质量 | 可能生成错误答案 |

## 四、为什么我要做这个项目？

### 背景
作为一名研二学生，我在写论文时经常遇到：
1. 综述阶段需要快速了解某个领域的研究现状
2. 现有的 AI 搜索工具要么太粗糙，要么太复杂
3. 很多工具会引入低质量甚至错误的信息

### 我的解决方案
基于对竞品的分析，我要做一个**平衡型 AI 搜索产品**：

**设计原则**：
1. **三层兜底机制**：Tavily → DuckDuckGo → 兜底结果，确保搜索稳定
2. **结果筛选**：验证搜索结果质量，过滤脏数据
3. **流式输出**：实时展示搜索进度，让用户看到 AI 的思考过程
4. **多轮对话**：支持澄清问题，逐步理解用户需求
5. **来源标注**：所有答案都标注来源，可追溯
6. **简洁 UI**：避免复杂交互，降低使用门槛

### 与竞品的差异化

| 维度 | 竞品 | 我的方案 |
|------|------|----------|
| 兜底机制 | 大部分无 | 三层兜底 + 重试 + 缓存 |
| 结果筛选 | 少数产品有（Wisper） | 有 |
| 流式输出 | 部分 | 有 |
| 多轮对话 | Kimi 有 | 有 |
| UI 简洁度 | 普遍复杂 | 极简设计 |
| 开源 | 大部分闭源 | 开源，可自部署 |

## 五、产品愿景

**短期**：做一个能用的 AI 搜索 Demo，展示核心流程

**中期**：优化搜索质量，加入学术数据库支持

**长期**：成为学生和研究人员做综述时的首选工具

## 六、技术选型

| 模块 | 技术方案 | 理由 |
|------|----------|------|
| 搜索 API | Tavily + DuckDuckGo | Tavily 质量高，DuckDuckGo 兜底 |
| AI 模型 | Moonshot/Kimi | 中文理解能力强，长上下文 |
| 前端 | React + Vite | 现代化、开发体验好 |
| 后端 | Node.js + Express | 轻量、易于部署 |
| 流式输出 | SSE (Server-Sent Events) | 实时性好，比 WebSocket 简单 |
